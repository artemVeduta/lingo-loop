# v0.1.1 Fast-Follow Handoff — `tutor doctor` broken on PyPI installs

Pick this up in a fresh session. Goal: ship `lingo-loop 0.1.1` that makes
`tutor doctor` honest for users who installed from PyPI (`uv tool install
lingo-loop`), then merge the held docs-flip PR #19.

## TL;DR

- `lingo-loop 0.1.0` is **published and immutable** on pypi.org (tag `v0.1.0`,
  GitHub Release live, build+publish workflow green). The package itself works:
  `uv tool install lingo-loop` installs the `tutor` CLI; `tutor init` materializes
  host plugin assets from the bundled wheel.
- **Defect:** `tutor doctor --json` reports **7 FAIL checks** for any non-editable
  (wheel) install — `manifest`, `setup_skill`, `vocab_skill`, `writing_skill`,
  `progress_skill`, `judge_agent`, `cli`. The handoff smoke "`tutor doctor` →
  `status: ok`" cannot pass for a real PyPI install.
- 0.1.0 cannot be re-cut (PyPI immutable) → fix ships as **0.1.1**.

## How it was found

Ephemeral smoke after publish:

```bash
uvx --refresh --from lingo-loop tutor doctor --json
```

Returned `ok` for runtime/dirs/yaml/sqlite, but `fail` for the 7 plugin-source
checks with repair hints like `Restore .claude-plugin/plugin.json`,
`Restore skills/tutor-setup/SKILL.md`, `Restore bin/tutor`.

## Root cause

`src/language_tutor/health.py:15-34` — `doctor()` resolves plugin assets against
`repo_root`:

```python
for name, rel in plugin_root_components().items():
    path = repo_root / rel          # health.py:26
```

`repo_root` comes from `src/language_tutor/cli.py:75-76`:

```python
def repo_root() -> Path:
    return Path(__file__).resolve().parents[2]
```

- **Editable/source install:** `parents[2]` == repo root, which has
  `.claude-plugin/`, `skills/`, `agents/`, `bin/` → checks pass.
- **Wheel install:** `__file__` is `…/site-packages/language_tutor/cli.py`, so
  `parents[2]` is a `site-packages`-adjacent lib dir with no plugin tree →
  all 7 checks FAIL.

`plugin_root_components()` (`src/language_tutor/adapters/claude.py:12-21`) returns
relative paths: `.claude-plugin/plugin.json`, `skills/tutor-{setup,vocab,writing,
progress}/SKILL.md`, `agents/tutor-judge.md`, `bin/tutor`.

## Why a naive redirect is not enough

There is already a correct resolver — `src/language_tutor/installer/assets.py`
`bundled_assets_root()` / `bundled_assets_root_for()` — which handles editable
**and** wheel installs (env override → repo-root sentinel `.claude-plugin/
plugin.json` → `importlib.resources.files("language_tutor") / "_assets"`).
`tutor init` already uses it. `doctor` should too — **but**:

- The wheel `force-include` (`pyproject.toml:70-79`) ships only the **manifests**
  + hermes-profile + openclaw payload into `language_tutor/_assets/`. It does
  **NOT** ship `skills/*/SKILL.md`, `agents/tutor-judge.md`, or `bin/tutor`.
- Verified against the published wheel:
  ```
  language_tutor/_assets/.claude-plugin/plugin.json
  language_tutor/_assets/.codex-plugin/plugin.json
  language_tutor/_assets/openclaw-plugin/{package.json,openclaw.plugin.json,tsconfig.json,src/index.ts}
  language_tutor/_assets/hermes-profile/{distribution.yaml,config.yaml,SOUL.md}
  ```
  No skills/agents/bin.

So pointing `doctor` at `bundled_assets_root()` fixes the **manifest** check
only; the skill/agent/cli checks still fail on wheels because those files are
not bundled.

## Design decision required (pick before coding)

What should `tutor doctor` verify for a PyPI user?

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| **A. Bundle + redirect** | Add `skills/`, `agents/`, `bin/tutor` to `force-include`; change `doctor` to resolve via `bundled_assets_root()` instead of `repo_root()` | All checks pass on wheels; keeps checks meaningful; matches existing host-package pattern | Grows wheel; must confirm the Claude plugin actually needs these payloads bundled (vs `tutor init` only writing `plugin.json`) |
| **B. Check installed host plugin** | `doctor` verifies the post-`init` host plugin location (e.g. `~/.claude/plugins/lingo-loop/`), not bundled source | Semantically correct (checks what the user actually runs) | Bigger change; must define behavior pre-`init` (no plugin yet → "not installed", not "fail") |
| **C. Relax for wheels** | `doctor` resolves manifest via `bundled_assets_root()`; treat skill/agent/cli as dev-only checks, reported `n/a`/skipped on non-editable installs | Smallest change; honest output | Loses end-user verification of plugin payload integrity |

Recommendation: **A** if the Claude plugin payload genuinely needs to ship in the
wheel (confirm what `tutor init` writes for Claude — install doc says it copies
only `plugin.json` to `~/.claude/plugins/lingo-loop/plugin.json`); otherwise **C**
(KISS/YAGNI — don't bundle payloads nobody installs). Decide by checking the
Claude installer provider + what a working Claude plugin requires on disk.

Investigation pointers:
- Claude install flow: `docs/install/claude.md` (Step 1 — `tutor init`).
- Installer provider: `src/language_tutor/installer/providers/claude.py`.
- Resolver to reuse: `src/language_tutor/installer/assets.py`.
- The doctor code to change: `src/language_tutor/health.py`,
  `src/language_tutor/adapters/claude.py`, `src/language_tutor/cli.py:75`.

## Release mechanics for 0.1.1

`main` currently sits at the `v0.1.0` release commit. Standard path:

1. Fix on a feature branch off `main` (TDD). Add a CHANGELOG `[Unreleased]` entry.
2. PR → merge to `main`.
3. `/pre-release` → must be clean.
4. `/release-cut 0.1.1` → opens `chore/release-0.1.1` PR (bumps `pyproject.toml`
   `0.1.0` → `0.1.1`, promotes CHANGELOG). Merge it.
5. `/release-tag 0.1.1` → tags `v0.1.1`, pushes → workflow publishes to PyPI.
   (`scripts/version-guard.sh` requires `pyproject.toml` == tag; the release-cut
   PR provides the bump.)
6. Smoke: `uvx --refresh --from lingo-loop==0.1.1 tutor doctor --json` → expect
   the chosen healthy output.

proven by the 0.1.0 publish — no PyPI account setup needed.
